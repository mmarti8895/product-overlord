# Proposal: prd-generation

## Problem

PMs must write PRDs, one-pagers, and release notes by hand, copying context from Jira, Confluence, and GitHub into a document template. The system already holds enriched canonical tickets, repo-confidence scores, and acceptance criteria generated by the readiness pipeline — none of that is ever surfaced as a publishable document. The result is significant PM toil, inconsistent formatting, and documents that go stale the moment the ticket changes.

## Intent

Add a **PRDWriter** module that uses the LLM layer and RAG context (existing embedding store) to draft PRDs, one-pagers, and release notes from enriched ticket data. Publishing to Confluence is gated behind explicit human confirmation, mirroring the existing Jira comment approval pattern (invariant 11).

## Scope

**In scope:**
- `PRDWriter` service: takes a `CanonicalTicket` or Epic key → produces a structured `PRDDraft` artefact
- Document templates: PRD (problem / goals / non-goals / user stories / acceptance criteria / risks), one-pager (problem / solution / success metrics), release note (what shipped / what changed / known issues)
- RAG retrieval: pulls related tickets, existing Confluence pages, and repo file summaries from LanceDB to ground the draft
- Human review step: draft displayed in UI; human edits inline before any publish action
- `POST /api/prd/:ticketKey/draft` — generate a new draft
- `POST /api/prd/:ticketKey/publish` — human-confirmed publish to Confluence (writes confluence page or creates a new child page under a configured parent)
- `GET /api/prd/:ticketKey/drafts` — list saved drafts for a ticket
- UI panel: `PRDPanel` with rich-text editor, diff view against existing Confluence page, and publish button
- `PRDDraft` type stored in LanceDB with version history

**Out of scope:**
- Autonomous publishing without human confirmation
- Generating PRDs for tickets below a readiness threshold (guard: `readiness_score < 0.4` → decline with suggestion to improve spec first)
- OKR or metric targets in the PRD (outcome-tracking capability provides that signal)
- Portfolio-level release plans spanning multiple projects

## Assumptions

- Confluence connection is already configured in `ConnectionManager` (`ConfluenceConfig`)
- LanceDB embedding store is queryable for RAG (existing `embed` and `upsert-lancedb` stages)
- LLM provider is configured and capable of structured output (JSON mode for template sections)
- Human review always happens in the desktop UI before any Confluence write is attempted
- Draft versions are persisted locally in LanceDB; they are NOT committed to Confluence until publish is confirmed

## Human Approval Points

- **Draft review**: every generated draft must be opened and explicitly approved in the UI before it is eligible for publishing
- **Publish confirmation**: a dedicated "Publish to Confluence" button triggers a confirmation modal listing the target page URL, parent page, and space key — no publish occurs without this confirmation
- **Diff display**: if a Confluence page already exists for the ticket, a side-by-side diff is shown before publish so the PM understands what will change

## Rollout Stage

Stage-10. Depends on Stage-9 (roadmap-planning) for epic context, Stage-6 (LLM enrichment) for the language model, and Stage-4 (embed/upsert-lancedb) for RAG retrieval.

## Rollback

Remove `PRDWriter`, the `/api/prd/*` routes, and the `PRDPanel`. Drafts stored in LanceDB are inert — removing the service leaves them orphaned but causes no data loss and does not affect any Confluence page already published.

## Non-Goals

- Not a full document management system
- Not replacing Confluence for storage — Confluence remains the system of record
- Not generating PRDs for every ticket automatically (on-demand only)
- Not enforcing a specific PRD template organisationally — templates are configurable

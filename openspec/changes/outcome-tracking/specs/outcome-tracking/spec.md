## ADDED Requirements

### Requirement: OKR management with epic linkage
The system SHALL support CRUD for `OKR` records via `/api/outcomes/okrs`. Each OKR SHALL have an objective, quarter, one or more key results, and an optional list of linked epic keys. Key results SHALL specify a `target`, `current` value, `unit`, and `direction` (`increase` | `decrease` | `maintain`). The system SHALL NOT autonomously create, close, or grade OKRs; all OKR lifecycle actions require a human API call.

#### Scenario: OKR created and linked to epic
- **WHEN** `POST /api/outcomes/okrs` is called with `{ objective: "Grow activation", linked_epic_keys: ["ONBRD-5"] }`
- **THEN** a new `OKR` SHALL be persisted to LanceDB and `GET /api/outcomes/okrs` SHALL include it

#### Scenario: Key result current value updated via PATCH
- **WHEN** `PATCH /api/outcomes/okrs/:id` is called with an updated `current` value for a key result
- **THEN** the stored key result SHALL reflect the new value and `updated_at` SHALL be refreshed

### Requirement: Metrics ingestion — LaunchDarkly adapter
When a `LaunchDarklyConfig` is present in `ConnectionManager`, the `LaunchDarklyAdapter` SHALL fetch flag evaluation data for flags tagged with `epic:{epicKey}`. It SHALL return `FlagAdoption` records including `evaluations_7d`, `on_percentage`, and a 7-day daily evaluation count array (`trend`). If LaunchDarkly is unreachable the system SHALL include `warnings: ['launchdarkly_unavailable']` in the `OutcomeSnapshot` and SHALL continue building the snapshot from other available signals.

#### Scenario: Flag adoption data retrieved
- **WHEN** `OutcomeSnapshotBuilder` processes epic `FEAT-10` and LaunchDarkly has a flag tagged `epic:FEAT-10`
- **THEN** the snapshot SHALL include a `FlagAdoption` entry with `flag_key`, `on_percentage`, and `trend` (7 values)

#### Scenario: LaunchDarkly unavailable — snapshot still created
- **WHEN** the LaunchDarkly API is unreachable during snapshot generation for epic `FEAT-10`
- **THEN** the snapshot SHALL be created with `flag_adoption: []` and `warnings` SHALL include `'launchdarkly_unavailable'`

### Requirement: Generic metrics webhook
`POST /api/outcomes/metrics/ingest` SHALL accept `{ source, metric_name, value, timestamp, dimensions }` authenticated with a Bearer token matching `ConnectionManager.webhookSecret`. Received events SHALL be persisted to LanceDB table `metric_events`. When `dimensions.epic_key` matches a known OKR key result with `source: 'webhook'` and matching `metric_name`, the `KeyResult.current` SHALL be updated automatically.

#### Scenario: Webhook updates key result current value
- **WHEN** a webhook event is received with `{ metric_name: "activation_rate", value: 42, dimensions: { epic_key: "ONBRD-5" } }` and an OKR key result for `ONBRD-5` has `metric_name = "activation_rate"` and `source = 'webhook'`
- **THEN** that key result's `current` SHALL be set to `42`

#### Scenario: Unauthenticated ingest rejected
- **WHEN** `POST /api/outcomes/metrics/ingest` is called without a valid Bearer token
- **THEN** the system SHALL return `401 Unauthorized` and SHALL NOT persist any event

### Requirement: Post-ship outcome snapshot
The system SHALL build an `OutcomeSnapshot` for an epic when all child tickets have transitioned to a configured done status. The snapshot SHALL include: OKR deltas (before/after per key result since ship date), feature flag adoption, and an LLM-generated `reflection_draft`. The `ReflectionAgent` SHALL compare acceptance criteria against measured outcomes to produce the draft. The snapshot `status` SHALL start as `'draft'` and transition to `'reviewed'` only when a PM saves reflection notes via `PATCH /api/outcomes/:epicKey/snapshot/notes`.

#### Scenario: Snapshot auto-triggered on epic completion
- **WHEN** the last child ticket of epic `FEAT-10` transitions to a done status (detected by `SprintMonitor`)
- **THEN** `OutcomeSnapshotBuilder` SHALL build and persist an `OutcomeSnapshot` with `status: 'draft'`

#### Scenario: Reflection draft generated
- **WHEN** an `OutcomeSnapshot` is built for epic `FEAT-10` with 2 key result deltas and 1 flag adoption record
- **THEN** `reflection_draft` SHALL be a non-empty Markdown string generated by the LLM

#### Scenario: PM notes flip status to reviewed
- **WHEN** `PATCH /api/outcomes/FEAT-10/snapshot/notes` is called with `{ notes: "Good result, will repeat." }`
- **THEN** the snapshot SHALL have `reflection_notes = "Good result, will repeat."` and `status = 'reviewed'`

### Requirement: RICE confidence feedback loop
After an `OutcomeSnapshot` is built, the system SHALL emit a confidence delta to `PrioritisationEngine`. Positive OKR deltas (at least one key result moved toward its target) SHALL boost RICE `confidence` for epics of the same labels by up to 10 percentage points. Negative deltas (all key results moved away from target) SHALL decay confidence by up to 15 percentage points. Neutral or missing deltas SHALL not change confidence.

#### Scenario: Positive outcome boosts RICE confidence
- **WHEN** an `OutcomeSnapshot` for an epic labelled `growth` has all positive OKR deltas
- **THEN** future RICE computations for epics labelled `growth` SHALL have their LLM-estimated confidence increased by up to 10 pp

#### Scenario: Negative outcome decays confidence
- **WHEN** all OKR deltas in an `OutcomeSnapshot` are negative
- **THEN** `PrioritisationEngine` SHALL decay the confidence for matching future epics by up to 15 pp

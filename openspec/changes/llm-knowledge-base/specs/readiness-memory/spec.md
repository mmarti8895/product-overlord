## MODIFIED Requirements

### Requirement: Reflection-driven profile updates
Readiness profiles SHALL be updatable via promoted reflection candidates only. A profile update SHALL record the source reflection candidate ID, the reviewer who approved it, and the timestamp of promotion.

#### Scenario: Profile updated after promotion
- **WHEN** a reflection candidate is approved by a reviewer
- **THEN** the readiness profile for the affected project+issue-type is updated with the new dimension weights
- **AND** the update is recorded with `{reflection_candidate_id, reviewer, promoted_at}`

## ADDED Requirements

### Requirement: LLM enrichment pass on readiness result
After the deterministic scorer produces a `ReadinessResult`, the system SHALL invoke `enrichReadiness(result, retrievedChunks, fileContents)` via the LLM adapter. The enrichment pass MAY add entries to `missing_items` with LLM-generated descriptions. It MAY add questions to `questions_for_pm`, `questions_for_engineer`, and `questions_for_qa` with justification strings. It SHALL NOT change `readiness_status` from `blocked`. It SHALL NOT reduce `readiness_score`.

#### Scenario: LLM adds missing items
- **WHEN** the LLM enrichment pass identifies a missing acceptance criterion not caught by keyword matching
- **THEN** the item is appended to `missing_items` with `{dimension, severity, reason, source: "llm"}`
- **AND** the deterministic `missing_items` entries are preserved unchanged

#### Scenario: LLM blocked verdict preserved
- **WHEN** the deterministic scorer returns `readiness_status: "blocked"`
- **THEN** the LLM enrichment pass runs but MUST NOT change the status to `ready` or `needs_clarification`

#### Scenario: LLM degraded — deterministic result returned
- **WHEN** the LLM adapter is in degraded mode (`DEGRADED_LLM=true` or API error)
- **THEN** the enrichment pass is skipped and the deterministic `ReadinessResult` is returned unchanged

### Requirement: Justified clarification questions
Clarification questions generated by the LLM enrichment pass SHALL include a `justification` string referencing specific retrieved context (KT chunk or code snippet) that motivated the question.

#### Scenario: Question with justification
- **WHEN** the LLM generates a clarification question for the PM persona
- **THEN** the question object includes `{persona, question, justification}` where `justification` is non-empty and cites a chunk
